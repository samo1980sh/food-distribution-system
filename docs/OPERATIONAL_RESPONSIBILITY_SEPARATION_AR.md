# فصل المسؤوليات التشغيلية بين لوحة الإدارة وتطبيق Flutter

## الهدف

تثبيت قاعدة تشغيلية واضحة تمنع تكرار العمل بين لوحة Filament والتطبيق الميداني:

- لوحة الإدارة: الإعداد، التخطيط، المراجعة، الاعتماد، الاستثناءات، الرقابة والتقارير.
- تطبيق Flutter: إنشاء العمليات الميدانية للمندوب والسائق والعمل Offline ثم المزامنة.

## التغييرات المطبقة

### مصدر العملية والتدقيق

أضيف الحقلان التاليان إلى:

- فواتير البيع.
- تحصيلات العملاء.
- مرتجعات المبيعات.
- مصاريف السيارات.
- الإغلاق اليومي.

الحقول:

- `operation_source`: مصدر العملية.
- `administrative_reason`: بيان أو سبب الإدخال الإداري.

مصادر العمليات المعتمدة:

- `mobile_sales`: تطبيق مندوب المبيعات.
- `mobile_driver`: تطبيق السائق.
- `office`: إدخال مكتبي.
- `admin_exception`: إدخال إداري استثنائي.
- `system`: النظام.
- `legacy`: بيانات سابقة قبل تطبيق الفصل.

العمليات القديمة التي تحتوي `client_reference` تُصنف تلقائيًا كعمليات ميدانية، وباقي السجلات السابقة تُصنف `legacy` دون تغيير قيمها المالية أو التشغيلية.

### صلاحيات إنشاء العمليات

- المندوب يحتفظ بصلاحيات إنشاء الفاتورة والتحصيل والمرتجع والمصروف عبر API فقط.
- السائق يحتفظ بصلاحية إنشاء مصروف السيارة عبر API.
- المشرف يراجع ويعدل المسودات ضمن صلاحياته، لكنه لا ينشئ العمليات الميدانية من لوحة الإدارة.
- المحاسب يستطيع تسجيل تحصيل مكتبي وإنشاء إغلاق إداري مؤقت.
- المدير وSuper Admin يستطيعان إنشاء فاتورة أو مرتجع أو مصروف إداري استثنائي مع سبب إلزامي.
- أوامر تحميل السيارات تبقى عملية إدارية في مجموعة التخطيط والتجهيز.

### واجهات Filament

نُقلت العمليات التالية إلى مجموعة **المراجعة والاعتماد**:

- فواتير البيع.
- تحصيلات العملاء.
- مرتجعات المبيعات.
- مصاريف السيارات.
- الإغلاق اليومي.

تظهر شارة بعدد العمليات التي تنتظر المراجعة أو الاعتماد ضمن نطاق المستخدم.

أصبحت أزرار الإنشاء واضحة الغرض:

- فاتورة إدارية استثنائية.
- مرتجع إداري استثنائي.
- تسجيل تحصيل مكتبي.
- مصروف إداري استثنائي.
- إغلاق إداري مؤقت.

كما يظهر مصدر العملية واسم منشئها في الجداول وصفحات التفاصيل.

### المخزون

أصبحت شاشة حركات المخزون بعنوان **سجل وتسويات المخزون**، وأصبح الإدخال اليدوي بعنوان **تسوية مخزون إدارية** مع سبب إلزامي. الحركات التشغيلية الناتجة عن الفواتير والمرتجعات وأوامر التحميل تبقى تلقائية كما هي.

## ما لم يتغير

- منطق الاعتماد والإلغاء.
- منطق المخزون والتكلفة.
- Push Batch وOutbox وidempotency.
- نطاقات الوصول.
- التقارير والطباعة والحسابات.
- أوامر تحميل السيارات.
- إنشاء الإغلاق من لوحة الإدارة مؤقتًا.

## الفحص اليدوي

1. حساب المشرف لا يرى أزرار إنشاء فاتورة أو مرتجع أو تحصيل أو مصروف أو إغلاق.
2. حساب المحاسب يرى تسجيل تحصيل مكتبي وإغلاق إداري مؤقت فقط.
3. حساب المدير يرى الإدخالات الاستثنائية ويُطلب منه سبب إلزامي.
4. الفاتورة القادمة من Flutter تظهر بمصدر **تطبيق مندوب المبيعات**.
5. مصروف السائق يظهر بمصدر **تطبيق السائق**.
6. التحصيل المكتبي يظهر بمصدر **إدخال مكتبي**.
7. أوامر تحميل السيارات تبقى ضمن **التخطيط والتجهيز**.
8. التسوية اليدوية للمخزون لا تُحفظ دون سبب واضح.
