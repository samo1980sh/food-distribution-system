# Mobile Operational Write API — Phase 1

## الهدف

تضيف هذه المرحلة واجهات كتابة تشغيلية آمنة لتطبيق Flutter فوق الأساس الموجود عند:

`c7e79eb Add mobile operational read API`

تعيد الواجهات استخدام Policies وRole Data Scopes وخدمات الترقيم والمخزون والحسابات والإغلاق اليومي الموجودة في النظام. لا يوجد منطق مخزون أو مالي مكرر داخل Controllers.

## النطاق

تدعم المرحلة إنشاء وتعديل وحذف المسودات أو السجلات المعلقة، ثم تنفيذ الإجراءات التي يسمح بها الدور الحالي:

- فواتير المبيعات: إنشاء، تعديل، حذف مسودة، اعتماد، إلغاء.
- تحصيلات العملاء: إنشاء، تعديل، حذف مسودة، اعتماد، إلغاء.
- مرتجعات المبيعات: إنشاء، تعديل، حذف مسودة، اعتماد، إلغاء.
- مصاريف السيارات: إنشاء، تعديل، حذف قيد المراجعة، اعتماد، رفض.
- الإغلاقات اليومية: إنشاء، تعديل، حذف مسودة، تحديث المجاميع، اعتماد، إلغاء.

جميع المسارات تقع تحت:

`/api/v1/operational`

## الأمان والصلاحيات

- كل طلب يحتاج Sanctum Bearer Token وقدرة `api:v1` وصلاحية `api.access`.
- الإنشاء والتعديل والحذف والإجراءات تمر عبر Laravel Policies الحالية.
- ScopedModelObserver يمنع حفظ Route أوVehicle أوWarehouse أوEmployee أوCustomer خارج النطاق الفعلي للمستخدم.
- Model binding يخفي السجلات خارج النطاق كـ404.
- لم تتم إضافة أي صلاحية أو توسيع أي دور في هذه المرحلة.
- مندوب المبيعات ينشئ ويعدل المستندات المسموح بها، لكنه لا يعتمدها ما لم يمتلك صلاحية الاعتماد.
- السائق ينشئ ويعدل مصاريف مركبته ضمن نطاقه، ولا يعتمدها.

## منع تكرار الإنشاء

كل طلب `POST` لإنشاء مستند يحتاج:

```json
{
  "client_reference": "device-01-invoice-20260716-0001"
}
```

يجب أن يكون المرجع ثابتاً وفريداً لكل مستخدم ونوع مستند. يخزن النظام أيضاً بصمة SHA-256 للبيانات:

- إعادة نفس المرجع مع نفس البيانات تعيد السجل نفسه مع `meta.idempotency.replayed = true` ولا تنشئ نسخة جديدة.
- إعادة المرجع نفسه مع بيانات مختلفة تعيد HTTP 409 ورمز `idempotency_conflict`.

هذه الحماية مخصصة لإعادة المحاولة الآمنة، وليست Offline Sync كاملاً. مرحلة Offline Queue والمزامنة ثنائية الاتجاه لم تنفذ بعد.

## حالات الاستجابة

- `201`: تم إنشاء سجل جديد.
- `200`: تحديث أو إجراء ناجح، أو إعادة سجل سابق لنفس `client_reference`.
- `401`: لا يوجد تسجيل دخول صالح.
- `403`: الدور أو النطاق لا يسمح بالعملية.
- `404`: السجل غير موجود أو خارج نطاق المستخدم.
- `409`: خرق قاعدة عمل، أو تعارض idempotency.
- `422`: بيانات الإدخال غير صالحة.

## قواعد المستندات

### فواتير المبيعات

- لا يقبل الـAPI رقم فاتورة أو حالة أو تكلفة مخزنية من العميل.
- يحسب `line_total` والمجاميع داخل Models وSalesInvoiceService.
- الاعتماد يستخدم InventoryMovementService ويطبق DailyClosingGuard.
- يمكن إدخال المنتج نفسه في تشغيلات أو صلاحيات مختلفة، ولا يمكن تكرار نفس مفتاح المنتج والتشغيلة والصلاحية.

### تحصيلات العملاء

- التحصيل المرتبط بفاتورة لا يقبل فاتورة خارج نطاق المستخدم، ويأخذ نطاقه التشغيلي من الفاتورة عند الحاجة.
- لا يمكن تحصيل أكثر من الرصيد المتبقي.
- التحصيل المستقل يحتاج مستودعاً عند الاعتماد.

### مرتجعات المبيعات

- المرتجع المرتبط بفاتورة لا يقبل فاتورة خارج نطاق المستخدم، ويتحقق من العميل والكميات المتاحة للإرجاع.
- تكلفة المرتجع لا تأتي من تطبيق الهاتف؛ تستعاد من الفاتورة الأصلية أو من بيانات المنتج حسب الخدمة الحالية.
- الاعتماد والإلغاء يمران عبر InventoryMovementService.

### مصاريف السيارات

- يدعم إيصالاً اختيارياً بصيغة JPG أو PNG أو WebP.
- الحد الافتراضي 5MB ويضبط عبر `MOBILE_API_EXPENSE_RECEIPT_MAX_KB`.
- عند استبدال الإيصال أو حذف السجل تنظف النسخة القديمة من public disk.

### الإغلاق اليومي

- الإنشاء ينشئ مسودة ثم يحسب المجاميع والمواد فوراً.
- تعديل `actual_cash_amount` أو نطاق الإغلاق يعيد حساب المجاميع.
- يمكن تحديث `actual_quantity` و`notes` للمواد الموجودة في ملخص الإغلاق فقط.
- لا يمكن إنشاء إغلاق فعّال مكرر لنفس التاريخ والمستودع.

## Bootstrap

أضيف إلى `/api/v1/operational/bootstrap`:

- `write.enabled`
- صلاحيات create/update/delete والإجراءات لكل وحدة.
- `idempotent_create`
- `client_reference_required`
- `sync.write_api_enabled = true`
- `sync.offline_queue_supported = false`

تظل مصفوفة `actions` داخل كل Resource هي المرجع النهائي للإجراءات المتاحة على السجل نفسه، لأن الحالة الحالية للسجل تؤثر في الصلاحية.
